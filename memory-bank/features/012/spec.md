# Spec: issue #12

## Источник
- GitHub issue: https://github.com/atamanuk/mapper/issues/12
- Заголовок: `Demo. Apply JsonPatch после мэппинга на реальный объект`

## Контекст
`Mapper.Demo` в `src/Mapper.Demo` сейчас покрывает только первую половину демонстрационного сценария: пользователь вводит `Json Patch` для `Mapper.Demo.Models.Source`, отправляет форму `Map`, а страница `src/Mapper.Demo/Pages/Index.cshtml` показывает сериализованный результат мэппинга в `JsonPatchDocument<Mapper.Demo.Models.Target>`.

Текущая серверная логика находится в `src/Mapper.Demo/Pages/Index.cshtml.cs`:
- принимает `PatchInput`;
- десериализует исходный patch через `Mapper.Demo.Services.PatchJsonSerializer.Deserialize<Source>()`;
- вызывает `_mapper.Map<Source, Target>(...)`;
- выводит mapped patch в `ResultJson`.

В issue требуется расширить этот demo-сценарий второй фазой: после получения mapped patch пользователь должен иметь возможность отдельно применить его к экземпляру целевой demo-модели и увидеть итоговое состояние объекта.

## Цель
Сделать в `Mapper.Demo` проверяемый пользовательский сценарий из двух последовательных действий:
1. получить `Json Patch` для `Target` через существующий шаг `Map`;
2. по отдельной кнопке `Apply` применить этот mapped patch к экземпляру `Mapper.Demo.Models.Target` и показать результат на той же странице.

## Что требуется реализовать
1. Расширить страницу `src/Mapper.Demo/Pages/Index.cshtml` и модель страницы `src/Mapper.Demo/Pages/Index.cshtml.cs`, чтобы кроме действия `Map` существовало отдельное пользовательское действие `Apply`.
2. Действие `Apply` должно работать с `Json Patch`, полученным после мэппинга в `JsonPatchDocument<Target>`, а не с исходным `JsonPatchDocument<Source>`.
3. Для сценария применения должен существовать детерминированный экземпляр `Mapper.Demo.Models.Target`, к которому и применяется mapped patch.
4. После выполнения `Apply` страница должна показать состояние экземпляра `Target` после применения patch в виде, пригодном для визуальной проверки пользователем.
5. Сценарий должен оставаться в рамках текущего demo-приложения `Mapper.Demo` и его единственной страницы `Index`, без добавления нового UI-потока вне существующей Razor Page.
6. Ошибки, возникающие при мэппинге или применении patch, должны оставаться наблюдаемыми на той же странице, без silent failure.

## Детализация поведения
1. При открытии `/` пользователь видит текущий demo-экран с вводом patch для `Source`.
2. После отправки `Map` приложение по-прежнему строит mapped patch для `Target` и показывает его пользователю.
3. Кнопка `Apply` доступна как отдельное действие относительно шага `Map`, а не как неявная часть мэппинга.
4. `Apply` не должен зависеть от неявного серверного состояния между HTTP-запросами: в рамках запроса на применение приложение должно однозначно получить mapped patch, соответствующий текущему значению `PatchInput`, и применить именно его.
5. Если для текущего `PatchInput` невозможно получить корректный mapped patch, `Apply` завершается видимой ошибкой и не показывает фиктивный результат применения.
6. Для применения используется новый экземпляр `Target` с явно зафиксированным начальным состоянием, одинаковым для каждого запуска сценария `Apply`.
7. Начальное состояние экземпляра `Target` должно отличаться от ожидаемого результата demo-патча хотя бы по `DisplayName` и `Age`, чтобы пользователь мог визуально увидеть изменение после применения.
8. Результат `Apply` отражает изменения в свойствах `Mapper.Demo.Models.Target`, которые затрагиваются mapped patch, включая уже существующий demo-мэппинг `Name -> DisplayName` и `Age -> Age`.
9. Представление результата должно явно показывать состояние целевой модели после применения patch, а не только повторно печатать список операций patch.

## Ограничения
- Источник требований ограничен issue `#12` и brief в `memory-bank/features/012/brief.md`.
- Использовать только фактические сущности текущего репозитория: `Mapper.Demo`, `Source`, `Target`, `IndexModel`, `PatchJsonSerializer`, `Mapper.Mapper`, `DemoProfile`.
- Спецификация не требует изменения публичного API библиотеки в `src/Mapper`; задача относится к demo-проекту.
- В issue не задан точный UI-дизайн, поэтому допустима любая реализация интерфейса в рамках `Index.cshtml`, если она сохраняет отдельное действие `Apply` и наблюдаемый результат применения.
- В issue не заданы детальные требования к текстам ошибок; важно лишь, чтобы ошибки были видимы пользователю и не терялись.
- Допустимо заново строить mapped patch внутри обработчика `Apply` из текущего `PatchInput`, если для пользователя сохраняется отдельный шаг `Map`, а в применение не попадает исходный `JsonPatchDocument<Source>`.

## Критерии приёмки
- В `src/Mapper.Demo/Pages/Index.cshtml` есть отдельное пользовательское действие `Apply` помимо существующего действия `Map`.
- После успешного `Map` пользователь может инициировать `Apply` без перехода на другую страницу.
- `Apply` использует mapped patch для `Target`, а не исходный patch для `Source`.
- `Apply` не требует наличия серверного in-memory состояния от предыдущего нажатия `Map`: повторяемый результат определяется текущим `PatchInput` и зафиксированным начальным состоянием экземпляра `Target`.
- После успешного `Apply` страница показывает итоговое состояние экземпляра `Mapper.Demo.Models.Target`.
- Для каждого запуска `Apply` используется одинаковое, явно определённое начальное состояние `Target`, отличное от ожидаемого итогового состояния demo-примера по полям `DisplayName` и `Age`.
- Для patch из текущего demo-примера итоговое состояние визуально подтверждает, что:
  - значение `Name` из source patch отразилось в `Target.DisplayName`;
  - значение `Age` отразилось в `Target.Age`.
- Если patch некорректен или применение невозможно, пользователь получает сообщение об ошибке на странице `Index`.
- Существующий сценарий отображения mapped patch не регрессирует: после `Map` пользователь по-прежнему видит сериализованный `JsonPatchDocument<Target>`.

## Уточнения для реализации
- Реализация должна явно зафиксировать начальное состояние экземпляра `Target` в коде demo и использовать его одинаково для каждого запуска `Apply`.
- Реализация должна определять mapped patch для `Apply` без опоры на неявное состояние предыдущего HTTP-запроса; допустимы повторный мэппинг текущего `PatchInput` или явная передача уже построенного mapped patch в форме.
- Если `JsonPatchDocument<Target>.ApplyTo(...)` вернёт ошибки модели, их нужно показать пользователю на странице `Index` отдельно от ошибок JSON-десериализации и ошибок мэппинга.
