# Plan: issue #12

## Шаг 1. Разделить серверную логику `IndexModel` на явные действия `Map` и `Apply`
Ожидаемый технический результат:
- В [src/Mapper.Demo/Pages/Index.cshtml.cs](/home/me/projects/mapper/src/Mapper.Demo/Pages/Index.cshtml.cs:1) появляются отдельные обработчики или эквивалентная явная ветвизация для действий `Map` и `Apply`.
- Логика `Map` по-прежнему принимает `PatchInput`, строит `JsonPatchDocument<Target>` через `PatchJsonSerializer` и `Mapper.Mapper`, и сохраняет сериализованный mapped patch для отображения.
- Логика `Apply` в рамках текущего запроса повторно получает mapped patch именно из текущего значения `PatchInput` или из явно переданного в форму эквивалентного значения, а не из неявного серверного состояния предыдущего запроса.
- В `IndexModel` появляются отдельные данные для отображения сериализованного mapped patch, результата применения `Target` и ошибок применения, чтобы успешный `Map` не подменял собой результат `Apply`.

## Шаг 2. Зафиксировать детерминированный сценарий применения patch
Ожидаемый технический результат:
- В [src/Mapper.Demo/Pages/Index.cshtml.cs](/home/me/projects/mapper/src/Mapper.Demo/Pages/Index.cshtml.cs:1) выделен явный способ создания стартового экземпляра `Mapper.Demo.Models.Target` для `Apply`.
- Начальное состояние отличается от ожидаемого результата demo-примера по `DisplayName` и `Age`, поэтому применение patch визуально меняет оба поля.
- Ошибки JSON-десериализации, ошибки мэппинга и ошибки `JsonPatchDocument<Target>.ApplyTo(...)` обрабатываются как наблюдаемые состояния страницы `Index`, без silent failure и без фиктивного результата применения.
- Ошибки `ApplyTo(...)`, включая ошибки модели, выводятся как отдельный вид результата применения и не маскируются под ошибки парсинга или мэппинга.

## Шаг 3. Обновить Razor Page под отдельный пользовательский шаг `Apply`
Ожидаемый технический результат:
- В [src/Mapper.Demo/Pages/Index.cshtml](/home/me/projects/mapper/src/Mapper.Demo/Pages/Index.cshtml:1) появляется отдельное пользовательское действие `Apply`, отправляющее тот же текущий `PatchInput` в собственном POST-запросе.
- Страница после `Map` по-прежнему показывает сериализованный `JsonPatchDocument<Target>`.
- После `Apply` та же страница явно показывает итоговое состояние объекта `Target` после применения patch в виде, удобном для визуальной проверки.
- Сообщения об ошибках для неуспешного `Map` или `Apply` остаются видимыми на странице `Index`, причём пользователь может отличить ошибку построения mapped patch от ошибки его применения.

## Шаг 4. Проверить сценарий страницы относительно требований spec
Ожидаемый технический результат:
- После успешного `Map` страница всё ещё показывает сериализованный `JsonPatchDocument<Target>`, а отдельный шаг `Apply` на той же странице показывает уже состояние экземпляра `Target`.
- Для одного и того же `PatchInput` повторный `Apply` даёт предсказуемый результат благодаря фиксированному начальному состоянию `Target`.
- Для невалидного `PatchInput` или ошибки применения страница показывает наблюдаемую ошибку вместо фиктивного результата.
