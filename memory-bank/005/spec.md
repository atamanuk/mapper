# Spec: issue #5

## Источник
- GitHub issue: https://github.com/atamanuk/mapper/issues/5
- Заголовок: `Формировать MapProfile.Definitions на этапе компиляции`
- Статус: `OPEN`

## Контекст
Сейчас `MapProfile` заполняет внутренний список `_definitions` в runtime: каждый вызов `CreateMap<TSource, TTarget>()` внутри конструктора конкретного профиля создаёт `MapDefinition<TSource, TTarget>` и добавляет его в коллекцию. Это означает, что описание мэппингов собирается при создании экземпляра профиля, а не заранее.

В issue требуется перенести формирование правил мэппинга в compile time с помощью Source Generators. Генератор должен анализировать код проекта-потребителя, находить профили мэппинга и генерировать готовое описание правил, которое затем используется runtime-компонентами без runtime-разбора builder-цепочек.

## Цель
Убрать runtime-формирование `MapProfile.Definitions` из fluent-builder-побочных эффектов и заменить его на compile-time генерацию эквивалентных описаний мэппинга:
- на этапе компиляции анализировать код текущего проекта;
- находить классы, наследующие `MapProfile`;
- извлекать из их конструкторов все конфигурации, начинающиеся с `CreateMap`;
- сохранять в исходном коде декларативное описание мэппинга через `CreateMap(...).ForMember(...)`;
- генерировать код, который регистрирует или материализует правила мэппинга без необходимости исполнять fluent-цепочки ради наполнения `Definitions`;
- использовать generated registry/factory для инициализации `Definitions` в самом `MapProfile`, а не подменять `Mapper` новым generated-типом;
- сохранить текущее поведение `Mapper`, `PatchMapCompiler` и существующих тестов без расширения test suite.

## Что требуется реализовать
1. Добавить в solution отдельный проект source generator.
   - Генератор должен подключаться ко всем проектам, в которых компилируются классы-наследники `MapProfile`.
   - Недостаточно подключить generator только к проекту библиотеки, если профили объявляются в других проектах solution.
2. Source generator должен анализировать только код компилируемого проекта, без загрузки внешних сборок с профилями и без runtime-reflection по assembly.
3. Генератор должен находить все непabstract-классы, наследующие `MapProfile`, и разбирать их конструкторы.
4. Внутри конструктора профиля генератор должен поддержать цепочки, начинающиеся с `CreateMap<TSource, TTarget>()`, включая последующие вызовы `ForMember(..., opt => ...)`.
5. Для каждого найденного `CreateMap` генератор должен восстанавливать полное описание `MapDefinition`, эквивалентное тому, которое сейчас собирается runtime-кодом:
   - пара `TSource -> TTarget`;
   - все `MemberConfiguration`, полученные через `ForMember`;
   - варианты `Ignore()`;
   - варианты `MapFrom(...)`.
6. Сгенерированный код должен использовать существующие runtime-типы (`MapDefinition`, `MemberConfiguration` и связанный API) либо новую внутреннюю модель хранения правил мэппинга, если она упрощает генерацию.
   - Допускается изменить внутреннее хранение `Definitions` и связанных структур.
   - Не допускается дублировать в generated-файлах runtime-логику `Mapper`/`PatchMapCompiler`; generated-code должен поставлять данные, а не новую реализацию мэппинга.
   - Не допускается решать задачу через генерацию отдельной реализации `IMapper`/`Mapper`, которая обходит `MapProfile.Definitions`.
7. Базовый `MapProfile` должен перестать зависеть от runtime-побочного эффекта вызова `CreateMap` для наполнения `Definitions`.
   - Вызовы `CreateMap`, `ForMember`, `Ignore`, `MapFrom` остаются декларативным DSL-описанием правил в исходном коде.
   - Исполнение этих вызовов в конструкторе профиля не должно быть обязательным для получения корректных `Definitions`.
8. Runtime API для прикладного кода должно остаться совместимым:
   - профили по-прежнему наследуются от `MapProfile`;
   - сигнатуры `CreateMap`, `ForMember`, `Ignore`, `MapFrom` не ломаются без необходимости;
   - `new SomeProfile()` по-прежнему даёт объект с корректно заполненными `Definitions`.
   - Пользователь не обязан объявлять профили как `partial` и не обязан менять стиль объявления профиля.
9. Если generator не смог корректно разобрать профиль или конкретный builder-сценарий, сборка должна завершаться диагностикой compile time, а не молча откатываться к старому runtime-наполнению.
10. Новые runtime/unit-тесты в рамках этой задачи не добавляются; функциональная корректность поведения мэппинга подтверждается прохождением уже существующих тестов.
11. Для неподдерживаемых generator-сценариев нужно явно определить фиксированный набор compile-time диагностик и минимальные требования к их содержимому:
   - идентификатор диагностики;
   - уровень (`error`);
   - краткое человекочитаемое сообщение;
   - имя профиля;
   - ссылка на проблемный вызов или выражение.
12. Спецификация должна явно разделять:
   - что подтверждается существующими runtime-тестами;
   - что подтверждается сборкой solution и наличием/отсутствием compile-time диагностик;
   - что проверяется код-ревью generated/runtime integration без добавления новых тестов.

## Ожидаемое поведение
- После компиляции каждый профиль `MapProfile` получает готовое compile-time описание мэппингов.
- При создании экземпляра профиля `Definitions` уже содержит все `IMapDefinition`, необходимые `Mapper`.
- В конструкторе профиля не требуется выполнять runtime-логику построения builder-объектов ради наполнения definitions.
- Декларативные вызовы `CreateMap(...).ForMember(...)` остаются в пользовательском коде как источник конфигурации.
- `Mapper` продолжает итерироваться по `profile.Definitions` и компилировать мэппинги без изменений пользовательского сценария.
- Инициализация generated-метаданных профиля происходит автоматически при загрузке assembly через generated registration code; пользовательский код не вызывает отдельный `InitMappings()`.
- Все существующие сценарии мэппинга продолжают работать:
  - same-name mapping;
  - rename через `MapFrom`;
  - `Ignore()`;
  - конвертация типов;
  - nested mapping, уже покрытый текущими тестами;
  - вычисляемые `MapFrom`, уже покрытые текущими тестами.
- Если профиль использует неподдержанный для generator синтаксис, разработчик получает понятную compile-time diagnostics с указанием профиля и проблемного выражения.
- При наличии нескольких конструкторов у одного профиля generator анализирует только тот конструктор, который реально будет выполнен при `new SomeProfile()` без параметров.
- Если у профиля нет доступного конструктора без параметров, такой профиль считается неподдержанным сценарием и сборка завершается compile-time диагностикой.

## Предлагаемая модель реализации
1. В библиотеке должна появиться явная точка расширения для generated-данных профилей: реестр вида `GeneratedProfileRegistry`, сопоставляющий `Type profileType` с фабрикой `IReadOnlyList<IMapDefinition>`.
   - Реестр является источником данных для базового `MapProfile`.
   - Решение не должно требовать, чтобы пользовательский профиль становился `partial`.
2. Базовый `MapProfile` при создании экземпляра должен получать `Definitions` из `GeneratedProfileRegistry.CreateDefinitions(GetType())`.
   - `Definitions` не должны больше наполняться runtime-побочным эффектом исполнения `CreateMap`.
3. Source generator должен генерировать отдельный internal-класс bootstrapper/registry для compilation-проекта.
   - Generated-класс должен регистрировать фабрики профилей автоматически через `ModuleInitializer`.
   - Пользовательский код не должен вызывать отдельную инициализацию вручную.
4. Для каждого найденного профиля generator должен генерировать factory-метод, возвращающий `IReadOnlyList<IMapDefinition>` для этого профиля.
5. Для каждого найденного `CreateMap` generator должен генерировать код materialization соответствующего `IMapDefinition`.
6. Если для представления `MapFrom` требуется сохранить expression tree, generator должен генерировать код, создающий эквивалентное `LambdaExpression`, совместимое с текущим `PatchMapCompiler`.
7. Generated-code должен использовать специальный facade API библиотеки для создания definitions и member-конфигураций.
   - Предпочтительный вариант: публичный generated-friendly builder/factory слой вроде `GeneratedMapping` и `GeneratedMapDefinitionBuilder<TSource, TTarget>`.
   - Generated-code не должен зависеть от прямого конструирования внутренних runtime-типов, если это требует излишнего раскрытия внутренностей библиотеки.
8. Генерируемый код должен быть детерминированным и не должен зависеть от порядка обхода синтаксического дерева, кроме стабильного порядка обнаруженных `CreateMap` в конструкторе.
9. Если для совместимости потребуется сохранить `CreateMap` и builder-объекты, они не должны повторно добавлять те же definitions во время выполнения.
10. Подключение generator в проекты-потребители должно быть частью поставки библиотеки как analyzer/reference integration, а не ручной локальной настройкой только для текущего solution.
   - Минимально допустимы два варианта: отдельный analyzer-пакет, который обязан подключить потребитель, либо автоматическая analyzer-интеграция через package/project reference библиотеки.
   - Для текущего solution должно быть реализовано фактическое подключение generator к `src/Mapper.Tests` как к проекту, где объявлены профили.
11. Поиск мест получения `IMapper` через DI, генерация отдельной реализации `IMapper` и подмена стандартного `Mapper` через DI не входят в выбранную архитектуру решения.

## Границы задачи
- В объём задачи входит compile-time генерация только для конфигураций, описанных в коде проекта через профили `MapProfile`.
- В объём не входит изменение публичной модели использования библиотеки для конечного кода.
- В объём не входит добавление новых функциональных возможностей мэппинга.
- В объём не входит расширение набора runtime/unit-тестов.
- В объём не входит fallback на runtime-сборку definitions, если generator по какой-то причине не отработал.
- В объём входит явная фиксация поддерживаемого поднабора синтаксиса профилей, который generator обязан разбирать.
- В объём входит проверка того, что generator реально подключён к проектам, содержащим профили в текущем solution.
- В объём не входит анализ DI-регистраций, поиск мест инъекции `IMapper` и генерация отдельного runtime-mapper-класса вместо существующего `Mapper`.

- [src/Mapper/Mapper.cs](/home/me/projects/mapper/src/Mapper/Mapper.cs) уже содержит всю runtime-логику компиляции `MapDefinition` в `ICompiledMap`; её желательно переиспользовать, а не переносить в generator.
- В solution сейчас нет проекта с Roslyn Source Generator, а [src/Mapper/Mapper.csproj](/home/me/projects/mapper/src/Mapper/Mapper.csproj) не содержит analyzer/reference-интеграции.
- Существующие тесты в [src/Mapper.Tests/MapperTests.cs](/home/me/projects/mapper/src/Mapper.Tests/MapperTests.cs) и [src/Mapper.Tests/NestedMapperTests.cs](/home/me/projects/mapper/src/Mapper.Tests/NestedMapperTests.cs) уже фиксируют текущее поведение мэппинга и должны остаться зелёными без добавления новых тестов.
- Профили в текущих тестах объявлены как обычные классы, а не `partial`; это ограничение нужно учитывать при выборе архитектуры генерации.

## Критерии приёмки
- `MapProfile.Definitions` заполняется compile-time сгенерированными готовыми значениями.
- Заполнение definitions происходит в конструкторе `MapProfile` или эквивалентной compile-time подключённой точке, без runtime-анализа builder-цепочек.
- Генератор находит все вызовы `CreateMap` в конструкторах классов-наследников `MapProfile` текущего проекта.
- Генератор корректно переносит в generated-code конфигурации `ForMember(...).Ignore()` и `ForMember(...).MapFrom(...)`.
- Генератор корректно переносит вычисляемые `MapFrom(...)`, совместимые с текущим runtime-поведением.
- При успешной генерации проект компилируется.
- Все существующие тесты проходят через `dotnet test`.
- Новые runtime/unit-тесты не добавляются.
- При неподдерживаемом синтаксисе профиля сборка падает с понятной compile-time диагностикой, а не с runtime-ошибкой после запуска.
- Не возникает дублей definitions из-за одновременной работы generated-кода и старого runtime-механизма.
- Пользовательский код профилей не требует модификации вида `partial class` или перехода на другой DSL.
- Для профилей текущего solution generator фактически запускается в тех проектах, где объявлены профили, а не только в проекте библиотеки.
- Generated registration code выполняется автоматически при загрузке assembly и регистрирует фабрики профилей без явного вызова из пользовательского кода.
- Стандартный `Mapper` остаётся единственной runtime-реализацией мэппинга; generated-code поставляет данные и bootstrap, а не заменяет `Mapper`.

### Матрица верификации
- Через существующие `dotnet test` подтверждаются: same-name mapping, rename через `MapFrom`, `Ignore()`, конвертация типов, nested mapping, вычисляемые `MapFrom`, отсутствие поведенческих регрессий `Mapper`.
- Через успешную сборку solution подтверждаются: подключение generator к проектам с профилями, успешная генерация кода, отсутствие необходимости объявлять профили как `partial`, доступность generated bootstrapper и registry registration.
- Через code review реализации подтверждаются: отсутствие fallback на старое runtime-наполнение `Definitions`, отсутствие дублирования runtime-логики `Mapper`/`PatchMapCompiler` в generated-code, отсутствие дублей `Definitions`, отсутствие подмены стандартного `Mapper` отдельной generated-реализацией.
- Через негативную сборку с compile-time диагностикой подтверждаются: реакция на неподдерживаемый синтаксис и качество сообщений диагностики.

## Архитектурная фиксация решения
- Источником истины для runtime-мэппинга остаются `MapProfile.Definitions` и существующий класс `Mapper`.
- Source generator анализирует только декларативный DSL профилей и генерирует:
  - bootstrapper с `ModuleInitializer`;
  - регистрацию factory-методов профилей в `GeneratedProfileRegistry`;
  - materialization кода для `IMapDefinition`.
- Базовый `MapProfile` получает definitions из generated registry по типу конкретного профиля.
- Generated-code не анализирует DI-контейнер, не ищет места инъекции `IMapper` и не генерирует отдельный mapper-класс с собственной логикой `Map<TSource, TTarget>()`.
- Если в библиотеке потребуется специальный API для generated-code, он должен быть оформлен как facade/factory слой, а не как раскрытие лишней runtime-логики наружу.

## Поддерживаемый generator-синтаксис
- Поддерживаются только прямые вызовы `CreateMap<TSource, TTarget>()` в теле конструктора профиля без промежуточного сохранения builder-а в переменную и без передачи builder-а в другие методы.
- Поддерживаются цепочки `CreateMap(...).ForMember(...).ForMember(...)`.
- Поддерживается `ForMember(target => target.Path, opt => opt.Ignore())`.
- Поддерживается `ForMember(target => target.Path, opt => opt.MapFrom(source => ...))`, если `target`-выражение и `source`-выражение представлены inline-лямбда-выражениями внутри того же вызова `ForMember`.
- Для `target`-лямбды поддерживается только прямой доступ к цепочке свойств от параметра:
  - `target => target.Member`
  - `target => target.Nested.Member`
- Для `source`-лямбды поддерживаются только формы, уже совместимые с текущим runtime API и покрытые существующими тестами:
  - прямой доступ к цепочке свойств: `source => source.Member`, `source => source.Nested.Member`;
  - вычисляемое выражение над ровно одной leaf-source веткой, если итоговое expression tree содержит ровно один исходный leaf-member path, из которого выводится patch-value;
  - вызовы стандартных методов и операторов внутри такого выражения допустимы только если generator может дословно воспроизвести исходное expression tree в generated-code без семантической трансформации.
- Не поддерживаются и должны диагностироваться как compile-time error:
  - `MapFrom`, где expression tree ссылается более чем на один source leaf-member path;
  - вынесенные helper-методы, локальные функции и method group вместо inline-лямбды;
  - предварительно сохранённые в переменные лямбды или builder-цепочки;
  - условные ветки и циклы на уровне построения DSL-конфигурации профиля;
  - динамическая сборка выражений, reflection и любой код, требующий исполнения конструктора для понимания конфигурации.
- Не поддерживаются условные ветки, циклы, локальные функции, helper-методы, промежуточные переменные builder-а, динамическая сборка выражений и иные формы, где конфигурация мэппинга не выражена прямой fluent-цепочкой в конструкторе.
- Любой синтаксис вне этого поднабора обязан приводить к compile-time ошибке generator-а.

## Каталог диагностик
- `MAPGEN001`, `error`: профиль не имеет доступного конструктора без параметров, который может быть использован как точка входа для декларативной конфигурации.
- `MAPGEN002`, `error`: generator встретил неподдерживаемую форму объявления `CreateMap`/`ForMember`, нарушающую разрешённый fluent-поднабор.
- `MAPGEN003`, `error`: `target`-лямбда в `ForMember` не является прямой цепочкой доступа к свойствам.
- `MAPGEN004`, `error`: `MapFrom` использует неподдерживаемое `source`-выражение, которое generator не может воспроизвести без изменения семантики.
- `MAPGEN005`, `error`: `MapFrom` зависит более чем от одного source leaf-member path.
- `MAPGEN006`, `error`: generator не может материализовать описание профиля из-за внутренне противоречивой конфигурации DSL.

Для каждой диагностики обязательны:
- ID диагностики;
- уровень `error`;
- имя профиля;
- текст, кратко описывающий причину;
- location проблемного синтаксического узла.

## Риски и вопросы
- Ключевой риск в том, что текущий fluent API исполняется как обычный C# код; генератору придётся распознавать ограниченный набор синтаксических форм, а не произвольную императивную логику в конструкторе.
- Допустимый поднабор профилей должен быть формально зафиксирован в спецификации; всё остальное должно явно диагностироваться.
- Поскольку целевая библиотека `netstandard2.1`, generator придётся вынести в отдельный проект с Roslyn-зависимостями и правильно подключить его так, чтобы generated-code был доступен и библиотеке, и тестовому проекту.
- Стоит отдельно проследить, чтобы generated-код имел доступ к internal-типам (`MapDefinition`, `MemberConfiguration`) без лишнего расширения публичной поверхности API, либо чтобы новая внутренняя модель хранения правил решала эту проблему чище.
